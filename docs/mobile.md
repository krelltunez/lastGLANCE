# Mobile apps (Capacitor)

lastGLANCE ships native Android and iOS shells built with
[Capacitor](https://capacitorjs.com/). Capacitor wraps the same Vite web build
(`dist/`) that powers the PWA inside a native WebView, so there is a single
codebase and the apps stay in lockstep with the web release.

## Identifiers

| Platform | Identifier |
|---|---|
| Android `applicationId` | `com.lastglance.app` |
| iOS bundle identifier | `com.lastglance` |
| App name | lastGLANCE |

The shared config lives in [`capacitor.config.ts`](../capacitor.config.ts).
The Android id is set there; the iOS id is overridden in
`ios/App/App.xcodeproj/project.pbxproj` (`PRODUCT_BUNDLE_IDENTIFIER`).

## Networking & sync

A native WebView enforces CORS just like a browser, which would block the
cross-origin WebDAV/Nextcloud requests that cloud sync depends on. In the
browser these go through a server-side CORS proxy (`VITE_WEBDAV_PROXY_URL`); in
the native apps that proxy is neither available nor needed.

Instead, on native the app makes WebDAV requests **directly** over the native
HTTP stack via [CapacitorHttp](https://capacitorjs.com/docs/apis/http), which is
not subject to the WebView's same-origin policy. Two things make this work:

1. **`capacitor.config.ts`** enables CapacitorHttp's global `fetch`/`XHR`
   patching as a safety net:

   ```ts
   plugins: { CapacitorHttp: { enabled: true } }
   ```

2. **Transport wiring** routes every WebDAV call through the native stack when
   `Capacitor.isNativePlatform()` is true (`src/sync/nativeHttp.ts`):
   - The `@glance-apps/sync` engine and providers receive an `electronProxyFetch`
     bridge backed by `CapacitorHttp.request`
     (`src/sync/engine.ts`, `src/utils/cloudSyncProviders.ts`). This bridge has
     priority over `proxyUrl`, so sync upload/download/backups bypass the proxy.
   - The app's own WebDAV helpers (`src/intents/webdav.ts`) — connection test,
     folder creation, backup browsing, and dayGLANCE intent delivery — call the
     target URL directly with a standard `Authorization` header.

   In the browser none of this changes: `isNativePlatform` is `false`, so the
   existing CORS-proxy path is used unchanged.

The Android `INTERNET` permission required for this is already present in
`android/app/src/main/AndroidManifest.xml`.

> **Service worker:** the Vite PWA service worker still ships in the bundle. On
> Android it registers over `https://localhost` and serves the precached assets;
> on iOS (the `capacitor://` scheme) it simply doesn't register. Either way the
> app's assets are bundled and served locally, so this is harmless — update
> checks just no-op inside the native shell.

## Building

The native projects are committed to the repo; the copied web assets and build
outputs are gitignored and regenerated by `cap sync`.

```bash
npm install            # installs Capacitor along with the web deps
npm run cap:sync       # builds the web app (dist/) and syncs into android/ + ios/
```

Then open the platform IDE:

```bash
npm run cap:android    # build + sync, then open Android Studio
npm run cap:ios        # build + sync, then open Xcode (requires macOS)
```

Or open them directly with the Capacitor CLI:

```bash
npx cap open android
npx cap open ios
```

### Requirements

- **Android**: a JDK (17+; the repo is verified with JDK 21) and the Android
  SDK command-line tools on `PATH` (`ANDROID_HOME`). **Android Studio is not
  required** — the build runs from the Gradle wrapper. `adb` (platform-tools)
  is needed only for the auto-install step.
- **iOS**: macOS with Xcode. Capacitor 8 uses Swift Package Manager, so no
  CocoaPods install is needed.

## Building the Android APK (CLI)

`build-android.sh` (ported from the dayGLANCE build script) wraps the whole
flow — web build, `cap sync`, and Gradle — so no IDE is needed:

```bash
./build-android.sh            # debug APK -> outputs/lastglance-debug.apk, then adb install
./build-android.sh --release  # signed release APK -> outputs/lastglance.apk
./build-android.sh --clean    # gradle clean + wipe dist/ first (combines with --release)
```

The debug build installs onto a connected device/emulator automatically (skipped
if none is attached). Build artifacts land in `outputs/` (gitignored).

`versionName`/`versionCode` are derived from `package.json` in
`android/app/build.gradle` (e.g. `1.8.4` → versionCode `10804`), so the native
version tracks the web app automatically.

### Release signing

Release builds are signed from `android/keystore.properties` (gitignored). Set
it up once:

```bash
# 1. Create a keystore (keep it safe — losing it means you can't update the app)
keytool -genkey -v -keystore android/lastglance-release.keystore \
  -alias lastglance -keyalg RSA -keysize 2048 -validity 10000

# 2. Configure signing (storeFile is relative to android/)
cp android/keystore.properties.example android/keystore.properties
# then edit android/keystore.properties with the real passwords/alias
```

Without `keystore.properties`, `--release` still builds but produces an
**unsigned** APK (not installable) — the script warns when this happens.

> AAB (Play Store bundle) and per-store flavors aren't wired up yet; add a
> `bundleRelease` step to `build-android.sh` when needed.

## Building for iOS

Build the web app, sync, and open Xcode to Archive:

```bash
npm run cap:ios          # build + cap sync ios + version sync + open Xcode
# or, without opening Xcode:
npm run build:ios
npx cap open ios         # then Product ▸ Archive in Xcode
```

(Signing uses your Apple Developer team in Xcode's *Signing & Capabilities*.)

### Version & build number

The iOS **marketing version** (`CFBundleShortVersionString`) is kept in sync
with `package.json` by `scripts/sync-ios-version.mjs`, which `build:ios` /
`cap:ios` run automatically (or run `npm run sync:ios-version` directly). It
rewrites `MARKETING_VERSION` in the Xcode project; the static `Info.plist`
references that via `$(MARKETING_VERSION)`, so Xcode substitutes it at build
time and never overwrites the plist.

The **build number** (`CURRENT_PROJECT_VERSION` / `CFBundleVersion`) must
*increment on every TestFlight/App Store upload* and is intentionally left
manual — bump it in Xcode (or with `agvtool new-version -all <n>`) per upload.

### Export compliance (encryption flag)

Unlike dayGLANCE (which generates its project from `project.yml` via XcodeGen),
lastGLANCE has a committed `.xcodeproj` and a static `Info.plist`. The
`ITSAppUsesNonExemptEncryption` key therefore goes **directly in
`ios/App/App/Info.plist`** — it is not overwritten by Xcode or `cap sync`.
Setting it avoids the encryption prompt on every TestFlight upload.

## Typical workflow

After any change to the web app, rebuild and re-sync before running natively:

```bash
npm run cap:sync
```

`cap sync` copies the latest `dist/` into both native projects and updates the
installed plugins. `build-android.sh` does this for you (`build:android`).
